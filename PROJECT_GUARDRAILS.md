# 项目开发边界（PROJECT_GUARDRAILS）

这个文件定义项目的开发边界：哪些不能动、哪些可以改、哪些改动必须先提案、哪些可以直接做、每次改完必须验证什么。

它不是为了限制开发，而是为了让每次改动都有明确的判断标准，不让项目在"顺手改一下"的过程中偏离方向。

---

## 一、不能动摇的（项目身份 / 底层方向）

这些是项目存在的根基，不能被修改。

### 1. 风控分析定位

项目是**赔率风控分析工具**，不是单纯预测比分的工具。

- 重点是判断：哪些比赛值得关注、哪些热门有风险、哪些盘口可能存在冷门信号、赛后复盘哪里判断错了。
- 不是："预测谁赢"的单一场景。
- 所有引擎和页面都要围绕风控判断来组织。

### 2. 先复盘 → 先模拟 → 先回测的验证方法

任何新规则 / 新权重 / 新数据字段，必须按以下流程验证：

1. **先复盘**：用已有的历史样本（2018 + 2022）分析问题出在哪，有多少场受影响。
2. **先模拟**：把新想法做成"模拟版"，和当前正式模型并列输出（类似 `engine/score-expander.js` 的做法）。
3. **先回测**：用历史样本比较"修改前 vs 修改后"的表现差异。
4. **再决定是否纳入正式模型**：只有在提案被同意后，才能把模拟升级为正式逻辑。

### 3. 影响正式模型前必须让我确认

正式模型、正式权重、正式评分规则的变更，不能单方面推送。必须先提交完整提案（见第三节），等你同意后再改。

### 4. 页面判断必须可解释

页面上每个判断（方向、比分区间、进球区间、风险评级）都要有可理解的依据：

- 不能只有数值结果而没有解释。
- 必须让普通用户能看懂"为什么这么判断"。
- 所有新增的判断逻辑，必须同时提供可解释的文本输出。

## 二、可以被打破和更新的（可变内容）

这些是实现细节，随时可以被改进，但**仍需按流程提案**（见第三节）。

| 分类 | 当前状态（可被挑战） | 相关文件 |
|---|---|---|
| 当前历史数据 | 16 场样本（2018 × 7，2022 × 9） | `data/matches-2018.js`、`data/matches-2022.js` |
| 当前评分权重 | `engine/config.js` 中的权重配置 | `engine/config.js` |
| 当前盘口/比分/进球规则 | scoring、intent、review 等引擎中的具体判断 | `engine/scoring.js`、`engine/intent.js`、`engine/review.js` |
| 当前页面结构 | 单页 HTML 入口 | `index.html` |
| 当前部署方式 | GitHub Pages + JSBox 离线版 | `DEPLOY.md` |

"可以被打破"的意思是：这些内容不是神圣不可改，但改之前必须提案并验证，不能顺手替换。

---

## 三、必须先提案才能改的（需要你同意）

只要涉及以下任何一项，必须先向你提交完整提案，等你同意后才能改代码：

### 提案格式

每次提案必须包含：

1. **问题**：当前哪里有问题（用具体的历史样本或回测结果说话）。
2. **证据**：具体数据——哪几场、命中率变化多少、覆盖率变化多少、回测对比是什么。
3. **建议**：具体怎么改——改哪个文件的哪段逻辑。
4. **好处**：改了之后模型在什么指标上会变好。
5. **风险**：改了之后可能引入什么新问题（过拟合、场景丢失、解释性下降、部署复杂化等）。

### 触发提案的具体场景

| 场景 | 说明 | 相关文件 |
|---|---|---|
| 正式模型逻辑变更 | 方向判断、风控评分、风险评级的核心逻辑 | `engine/scoring.js`、`engine/backtest.js` |
| 正式权重变更 | 各评分模块的权重分配 | `engine/config.js` |
| 数据字段结构变更 | 新增/删除/修改比赛数据字段 | `data/tournaments.js`、`data/matches-*.js` |
| 线上部署方式变更 | GitHub Pages 配置、部署流程变更 | `DEPLOY.md`、`index.html` |
| 真实数据源接入 | 从静态数据升级为自动拉取 | 任何新增数据接入代码 |
| 正式推荐规则变更 | 把"模拟版"逻辑升级为"正式版" | 任何涉及 finalPrediction / bettingReferencePlan 的改动 |

## 四、可以直接做的（不影响正式模型）

以下改动不需要先提案，可以直接做，但仍需符合每次开发后的验证要求（见第五节）：

| 类型 | 举例 |
|---|---|
| 新增模拟/对比模块 | 新增 `engine/score-expander.js` 这样的"对比展示"模块，不碰正式推荐 |
| 页面展示优化 | 调整 HTML 排版、样式文案、表格字段顺序，不改变底层判断逻辑 |
| 新增文档 | 新增或更新 `docs/` 下的说明文档 |
| 新增历史样本 | 按现有字段结构补充 2018 / 2022 或其他赛事的比赛数据（字段不变） |
| 代码结构整理 | 拆分模块、提取公共逻辑、调整文件组织，不改变行为 |

判断标准：**如果改动后，页面上的"正式推荐"没有变化，就不需要提案。**

## 五、每次开发完成后必须验证的

每次开发完成（包括模拟模块、展示调整、正式模型变更）后，必须确认以下全部通过才能提交推送：

### 5.1 文件层面

- [ ] 所有 `.js` 引擎文件语法正确（`node --check` 能通过）。
- [ ] `index.html` 能正常引入所有外部脚本，页面能正常打开。

### 5.2 功能层面

- [ ] 赛事总览正常展示，切换届别正常。
- [ ] 跨届汇总看板正常展示，各届汇总数据正确。
- [ ] 2022 回测仪表盘正常展示，命中率、比分覆盖、进球命中、冷门预警数字一致。
- [ ] 模型复盘建议正常展示。
- [ ] 模型版本对比正常展示。
- [ ] 如果本次改动涉及新增功能（如比分扩展模拟），新增区域能正常显示，没有破坏已有区域。

### 5.3 逻辑层面

- [ ] 本次改动前的正式推荐输出，和改动后的正式推荐输出是否一致？
- [ ] 如果不一致，必须在提案中说明并获得你同意。

### 5.4 文档层面

- [ ] `docs/DEVELOPMENT_LOG.md` 已追加本次操作记录。
- [ ] 如果新增了引擎模块，`README.md` 的文件说明表已更新。
- [ ] 如果变更了开发流程，`TRAE_CODE_BOOTSTRAP.md` 已更新。

---

## 六、一句话总结

> **模拟和对比可以自由做；把模拟升级为正式推荐，必须先问你。**
