# 项目底层逻辑和开发边界

这个文件是给 TRAE Code 看的。后续开发必须遵守这里的规则。  
如果某个改动触碰到“必须先征得用户同意”的范围，必须先停下来说明原因，等用户明确同意后再改。

## 项目根本目标

本项目不是一个单纯预测比分的工具，也不是为了给出“必胜结论”。

本项目的核心目标是：

```text
用赔率、盘口、水位、波胆和历史回测，帮助用户从风控角度判断比赛风险。
```

它要回答的是：

- 哪些比赛值得重点跟踪
- 哪些比赛需要等临场确认
- 哪些热门方向可能过热
- 哪些盘口有防平、防冷信号
- 模型赛后错在哪里
- 新规则是否真的比旧规则更稳

## 不能偏离的方向

后续开发不能把项目改成：

- 单纯比分预测软件
- 单纯胜平负预测器
- 娱乐竞猜页面
- 投注推荐系统
- 自动下注工具
- 只追求命中率、不看风险的模型

页面可以展示方向、比分区间和风险建议，但表达必须保持“风控分析”定位。

## 开发总原则

每一步开发必须遵守：

```text
先复盘，再开发。
先模拟，再替换。
先回测，再下结论。
```

任何新规则都不能直接进入正式模型。

正确流程是：

1. 先说明为什么要做这个改动
2. 先做模拟模块或对比模块
3. 用 2018 和 2022 历史样本回测
4. 页面展示新旧对比
5. 如果结果明显更好，再建议是否纳入正式模型
6. 是否纳入正式模型必须由用户确认

## 什么不能动摇

以下内容是项目底层方向，不能被改掉。

### 风控定位不能动摇

项目不是为了给用户一个“必中答案”。  
项目必须始终围绕：

```text
风险识别、盘口解释、赛后复盘、模型验证
```

可以预测方向和比分区间，但表达必须服务于风控判断。

### 验证方法不能动摇

任何规则都不能只靠感觉、单场案例或主观判断成立。

必须使用：

```text
历史样本
回测结果
新旧对比
赛后复盘
```

来判断它是否值得进入正式模型。

### 用户确认权不能动摇

如果改动会影响正式模型、正式结论、数据结构或线上部署，必须先问用户。

TRAE Code 可以提出建议，但不能替用户做最终决定。

### 可解释性不能动摇

页面不能只给数字。  
每个重要判断都要尽量解释：

- 为什么是这个方向
- 盘口和水位说明了什么
- 热门是否过热
- 为什么要防平或防冷
- 如果模型错了，可能错在哪里

普通用户也应该能看懂核心判断。

## 什么可以被打破和更新

以下内容不是绝对正确，可以被质疑、优化甚至替换。

### 当前数据可以被更新

现在的 2018 和 2022 样本是阶段性样本，不代表完整真理。  
后续可以继续补充、修正或替换样本。

但必须遵守：

- 新数据要符合当前字段结构
- 数据来源或录入逻辑要写清楚
- 新增数据后要重新跑回测
- 如果数据修正导致旧结论变化，要说明变化原因

### 当前权重可以被挑战

当前评分权重只是第一版经验规则，不是最终答案。

可以提出新的权重方案，但必须先作为模拟版本存在，例如：

```text
当前模型 vs 建议版模型
```

只有在跨届样本中表现更好，并且副作用可以接受时，才可以建议进入正式模型。

### 当前规则可以被替换

盘口意图、比分候选、进球区间、冷门预警等规则都可以优化。

但必须先走：

```text
新增模拟模块 → 历史回测 → 页面展示对比 → 用户确认 → 再进入正式逻辑
```

不能直接覆盖原规则。

### 当前页面结构可以优化

页面布局、移动端体验、图表、表格、说明文字都可以优化。

但不能牺牲：

- 页面可读性
- 风控定位
- 历史回测展示
- GitHub Pages 正常访问

### 当前部署方式可以升级

GitHub Pages 是当前阶段的部署方式，不代表永远不变。

后续如果 2026 真实数据需要自动更新，可以讨论：

- GitHub Pages + JSON 数据
- Cloudflare Pages
- Vercel
- 后端 API
- 数据库

但只要涉及部署方式变化，必须先向用户说明成本、好处和风险。

## TRAE Code 应该怎样提出突破性改动

如果 TRAE Code 认为某个底层规则、权重或数据结构需要改变，不要直接改。

应该按这个格式先告诉用户：

```text
我发现当前规则可能存在问题：
问题是什么：
证据是什么：
建议怎么改：
先做模拟还是直接改：
可能好处：
可能风险：
需要你确认后我再继续。
```

用户同意后才能进入实现。

## 必须先问用户的改动

以下改动不能直接做，必须先问用户。

### 修改正式评分权重

包括但不限于：

```text
engine/config.js
SCORING_CONFIG.weights
SCORING_CONFIG.riskThresholds
```

如果要修改风控分权重、重点跟踪阈值、等待确认阈值、建议过滤阈值，必须先说明：

- 为什么要改
- 当前问题是什么
- 新权重是什么
- 用历史样本回测后是否更好
- 可能带来什么副作用

用户同意后才能改正式配置。

### 修改正式推荐逻辑

包括但不限于：

```text
finalPrediction()
riskControlScore()
handicapProfile()
handicapRationality()
bettingReferencePlan()
```

这些函数会影响页面正式结论。  
不能因为某一次模拟结果更好，就直接替换正式逻辑。

### 删除或重写核心模块

不能直接删除或重写：

```text
engine/scoring.js
engine/backtest.js
engine/intent.js
engine/review.js
engine/model-advice.js
engine/model-version.js
engine/cross-tournament.js
engine/validate.js
```

如果确实需要重构，必须先说明：

- 为什么现有模块不能继续维护
- 重构范围有多大
- 哪些函数会受影响
- 如何验证结果没有变坏

### 修改数据字段结构

不能随意修改比赛数据字段。

当前核心字段包括：

```text
id
date
stage
home
away
actual
finalScore
predict
confidence
resultCorrect
probs
goals
scoreOdds
scoreProb
handicap
news
```

如果要新增字段可以先作为可选字段。  
如果要删除、重命名或改变字段含义，必须先问用户。

### 改变线上部署方式

当前线上部署方式是：

```text
GitHub Pages
https://uiyes.github.io/worldcup-odds-risk-control/
```

不能直接切换到其他部署方式。  
如果要引入 Vercel、Cloudflare、后端服务、数据库或 API，必须先问用户。

### 引入真实赔率数据源

如果要接入真实赔率、盘口、波胆数据源，必须先问用户。

需要说明：

- 数据从哪里来
- 是否需要账号
- 是否需要付费
- 是否有调用限制
- 是否会影响 GitHub Pages 静态部署
- 是否涉及合规风险

## 可以直接做的改动

以下改动可以直接做，但仍然要更新文档并验证页面。

### 新增模拟模块

例如：

```text
engine/score-expander.js
engine/goal-range-simulator.js
```

前提是：

- 不替换正式模型
- 页面明确标注为模拟或对比
- 用历史样本回测

### 新增页面展示

可以新增：

- 对比看板
- 复盘表格
- 数据检查提示
- 模型建议说明
- 手机端布局优化

前提是不能破坏已有页面。

### 补充历史样本

可以补充 2018、2022 历史比赛样本。  
但新增样本必须符合当前字段结构，并通过数据校验。

### 补充文档

可以直接补充：

```text
README.md
TRAE_CODE_BOOTSTRAP.md
docs/DEVELOPMENT_LOG.md
docs/2026_DATA_TEMPLATE.md
docs/TRAE_CODE_HANDOFF.md
```

## 每次开发完成必须做的事

每次开发完成后，必须检查：

- 页面能否正常打开
- GitHub Pages 是否会受影响
- `index.html` 是否正确引用新文件
- 新增 JS 文件是否语法正常
- 2018 和 2022 样本是否还能回测
- README 是否更新
- 开发记录是否更新
- 是否需要更新 `TRAE_CODE_BOOTSTRAP.md`

完成后必须告诉用户：

- 改了哪些文件
- 新增了哪些功能
- 是否改了正式模型
- 回测结果有什么变化
- 是否已经提交并推送到 GitHub
- 用户需要检查哪个网页位置

## 当前下一步优先级

当前最建议做的是：

```text
比分覆盖优化雏形
```

但只能作为模拟模块做，不能直接替换正式比分推荐。

建议新增：

```text
engine/score-expander.js
```

目标：

- 扩展相邻比分
- 扩展平局比分
- 扩展冷门比分
- 扩展大比分或其他比分分组
- 对比扩展前后的比分覆盖率
- 在页面展示模拟结果

## 给 TRAE Code 的硬性要求

如果不确定某个改动是否触碰核心逻辑，先问用户。

如果改动会影响正式模型判断，先问用户。

如果改动会影响线上部署方式，先问用户。

如果改动会改变数据字段结构，先问用户。

如果只是新增模拟模块、补充展示、更新文档，可以继续做，但必须完成验证。
